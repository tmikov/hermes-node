### R7. Implement minimal `contextify` binding -- ContextifyScript
**Depends on**: nothing
**What**: The CRITICAL missing binding. `vm.js` and `internal/vm.js` both require `internalBinding('contextify')` at load time. The binding must provide:
- `ContextifyScript` constructor class (called with `new ContextifyScript(code, filename, lineOffset, columnOffset, cachedData, produceCachedData, parsingContext, hostDefinedOptionId)`)
- `ContextifyScript.prototype.runInContext(sandbox, timeout, displayErrors, breakOnSigint, breakFirstLine)` method
- `makeContext` function
- `constants` object (with `measureMemory.mode.SUMMARY/DETAILED`, `measureMemory.execution.DEFAULT/EAGER`)
- `measureMemory` function
- `startSigintWatchdog` / `stopSigintWatchdog` functions
- `compileFunction` function (used by `internal/vm.js`)

**Action**: Create `lib/bindings/node_contextify.cpp` and `include/hermes/node-compat/bindings/node_contextify.h`.

The ContextifyScript class:
- **Constructor**: Accept all 8 args. Store the source `code` string and `filename` as persistent refs on the JS object (via `napi_set_named_property` with internal symbol keys or via `napi_wrap` data). Ignore `cachedData`/`produceCachedData`/`parsingContext`/`hostDefinedOptionId` for now.
- **`runInContext` method**: When `sandbox` is null/undefined (the `runInThisContext` case): evaluate `code` using `napi_run_script`. Set `//# sourceURL=filename` for stack traces. Return the result. When `sandbox` is an object: throw "vm.runInContext not implemented" for now.
- **`timeout` handling**: Ignore initially (timeout=-1 means no timeout). Can be added later.
- **`breakOnSigint`**: Ignore initially (stub in R8).

The class must be defined via `napi_define_class` and the resulting constructor stored so `vm.js` can do `class Script extends ContextifyScript`.

**Implementation detail**: `napi_run_script` evaluates code in the global context with `strict=false`, which matches `runInThisContext` semantics. The source URL can be set by appending `\n//# sourceURL=filename` to the code string.

**Success**: The following works:
```js
const vm = require('vm');
const result = vm.runInThisContext('1 + 2');
assert.strictEqual(result, 3);
const script = new vm.Script('x = 10');
script.runInThisContext();
assert.strictEqual(x, 10);
```
Test file: `test/test-vm-basic.js`.

### R9. Stub `makeContext` and `compileFunction` in contextify
**Depends on**: R7
**What**: `vm.js` exports `createContext` which calls `makeContext`. `repl.js` line 216 calls `vm.runInNewContext(...)` which calls `createContext` then `runInContext`. Also `internal/vm.js` uses `compileFunction` for `internalCompileFunction`.
**Action**:
- `makeContext(contextObject, name, origin, strings, wasm, microtaskQueue, hostDefinedOptionId)`: Set a marker property (using the `contextify_context_private_symbol` from the symbols binding) on `contextObject` and return it. - `constants`: object with `{measureMemory: {mode: {SUMMARY: 0, DETAILED: 1}, execution: {DEFAULT: 0, EAGER: 1}}}`.
- `measureMemory`: stub that returns `undefined`.
**Success**: `vm.runInNewContext('1+1')` returns `2`. `vm.createContext({})` returns an object where `vm.isContext(obj)` is `true`. `vm.compileFunction('return a+b', ['a','b'])(1,2)` returns `3`.

### R19. Implement SIGINT watchdog
**Depends on**: R7, R8
**What**: Replace the SIGINT stubs with real implementation so Ctrl+C interrupts long-running eval in the REPL.
**Action**: In `node_contextify.cpp`:
- `startSigintWatchdog()`:
  - Register a `uv_signal_t` handle (or POSIX `sigaction`) for SIGINT on a background thread or on the main loop.
  - On SIGINT, call `runtime->triggerTimeoutAsyncBreak()` (Hermes's `triggerAsyncBreak(AsyncBreakReasonBits::Timeout)`). This causes `notifyTimeout()` which raises a JS exception.
  - Requires access to the Hermes `Runtime*`. Use the `napi_get_instance_data` pattern or a global pointer (since there's only one runtime).
  - Hermes needs `asyncBreakCheckInEval` enabled in `RuntimeConfig` for async breaks to work during eval.
  - Return `true`.
- `stopSigintWatchdog()`:
  - Remove the SIGINT handler.
  - Return `true` if a SIGINT was received since `startSigintWatchdog()`, `false` otherwise.
- Ensure the RuntimeConfig in `hermes-node.cpp` enables `asyncBreakCheckInEval`.

**Implementation considerations**:
- Hermes `triggerTimeoutAsyncBreak()` is thread-safe.
- The timeout raises `Error: Javascript execution has timed out`. The REPL catches this and reports it.
- Alternative: use `triggerAsyncBreak` with a custom reason bit and handle it in a callback. But `Timeout` is the simplest path since `notifyTimeout()` already raises an exception.
- The signal handler must be lightweight (async-signal-safe). Just set the flag via `triggerTimeoutAsyncBreak()`.

**Success**: Start REPL, type `while(true){}`, press Ctrl+C -- execution is interrupted, REPL continues. Test:
```
// RUN: echo 'while(true){}' | timeout 2 %hermes-node 2>&1; test $? -ne 0
```
(Or a more sophisticated test using child_process to send SIGINT.)

## Risk Assessment

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| `repl.js` has unexpected deep dependency | Medium | Create targeted shims as discovered |
| `domain.js` needs working `async_hooks.createHook` | Medium | Our async_hooks stubs return no-ops; domain uses createHook for context tracking which is optional |
| `internal/modules/helpers.js` has cascading deps | Medium | Shim the module with just the functions REPL needs |
| `napi_run_script` strict-mode behavior differs from V8 eval | Low | `napi_run_script` already uses `compileFlags.strict = false` |
| Hermes `asyncBreakCheckInEval` not enabled | Low | Verify RuntimeConfig flag is set in hermes-node.cpp |
| `readline` raw mode doesn't work in all terminals | Low | Works via libuv's `uv_tty_set_mode`, well-tested |
