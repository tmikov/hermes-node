# Plan: Match Node's HTTP Parser ConnectionsList Behavior 1:1

## Background

`http.Server` in Node maintains a `ConnectionsList` of currently-attached
parsers. The list is the authoritative source for three server-level
operations:

1. `server.closeAllConnections()` â€” destroy every socket, idle or busy.
2. `server.closeIdleConnections()` â€” destroy only sockets that are not
   in the middle of parsing a request.
3. `server.headersTimeout` / `server.requestTimeout` â€” periodic
   `checkConnections()` runs `list.expired(headersTimeout, requestTimeout)`
   to find parsers stuck past their deadlines and force-close them.

In Node, these features are implemented natively in
`src/node_http_parser.cc`. Each `Parser` holds a `connectionsList_`
pointer and Push/Pop's itself at well-defined lifecycle points. The list
maintains *two* sets â€” a main set (everything) and an "active" subset
(parsers currently mid-request). `last_message_start_` (a `uv_hrtime()`
timestamp) lets `expired()` evaluate timeouts against wall time.

`hermes-node` ports the JS side (`libjs-node/_http_server.js`,
`libjs-node/net.js`) verbatim, but the binding (`lib/bindings/
node_http_parser.cpp`) was originally landed in commit
`9e857ca918dd "Step N5.15: Port http_parser binding"` with the
ConnectionsList class deliberately stubbed:

```cpp
// idle() and active() return all parsers (simplified â€” no native
// timeout tracking in our implementation).
napi_value idle(napi_env env)   { return all(env); }
napi_value active(napi_env env) { /* empty */ }
napi_value expired(...)         { /* empty */ }
```

â€¦and `ParserInitialize` simply ignored the `connectionsList` argument
that the JS side passed in (`args[4] = connectionsList (optional, unused
for now)`). `ParserRemove` was a no-op. The result is that the
*entire* connection-tracking machinery on the JS side runs against an
always-empty list, which makes:

- `closeAllConnections()` a silent no-op,
- `closeIdleConnections()` either a no-op (current behavior) or, if we
  partially fix it without modeling the active set, an over-eager
  destroyer that kills mid-request connections,
- header/request timeouts non-functional.

The first two are observable defects. The third is silent: requests
that should be terminated for taking too long simply continue.

## How did the gap happen?

The port was done as a single large step (~1300-line commit,
auto-generated by an agent) that focused on getting the parser path
working end-to-end. The JS side never actually stresses
`closeAllConnections()` in any of the existing http tests
(`test/test-http.js`'s keep-alive test 4 closes from the client agent
side, not via `closeAllConnections`), so the gap was not exercised.
This points to a methodology gap: when porting a binding from Node we
should test every public surface of the JS API, not only the happy path
through which our own usage flows. (See the "Tests" section below for
the specific cases we want to enforce going forward.)

## Specification (what Node does)

Numbers below refer to `~/3rd/node/src/node_http_parser.cc` at tag
`v24.13.0`.

### Parser member state

```cpp
ConnectionsList* connectionsList_;
uint64_t         last_message_start_;
```

### Lifecycle hooks

| Site | Operation |
|------|-----------|
| `Initialize`           (line 681-695) | `connectionsList_ = list; last_message_start_ = uv_hrtime(); list->Push(this); list->PushActive(this);` |
| `on_message_begin`     (line 269-287) | `Pop`+`PopActive`, `last_message_start_ = uv_hrtime()`, `Push`+`PushActive` |
| `on_message_complete`  (line 493-507) | `Pop`+`PopActive`, `last_message_start_ = 0`, `Push` only (parser stays in main list, leaves active subset â†’ idle) |
| `Remove` (binding)     (line 586-594) | `Pop`+`PopActive` |
| `~Parser`              (line 582)     | implicitly via `Remove` invariant |

The Pop-then-Push-with-fresh-timestamp idiom on `on_message_begin` is
*deliberate*: `ConnectionsList::all()` is a `std::set` keyed on
`(last_message_start_, parser_ptr)`. Mutating the timestamp without
re-inserting would corrupt the set's ordering. Node calls this out
explicitly:

```cpp
// Important: Pop from the lists BEFORE resetting the last_message_start_
// otherwise std::set.erase will fail.
```

### ConnectionsList

```cpp
class ConnectionsList : public BaseObject {
  std::set<Parser*, ParserComparator> all_connections_;
  std::set<Parser*, ParserComparator> active_connections_;

  void Push(Parser*);
  void Pop(Parser*);
  void PushActive(Parser*);
  void PopActive(Parser*);

  // JS-callable:
  static void All(args)          // -> Array<Parser>     (= all_connections_)
  static void Idle(args)         // -> Array<Parser>     (all_ \ active_)
  static void Active(args)       // -> Array<Parser>     (= active_)
  static void Expired(args)      // -> Array<Parser>     (see below)
};
```

`Expired(headersTimeout, requestTimeout)` walks `all_connections_`
sorted ascending by `last_message_start_`. For each parser:

- If `headers_completed_` is true â†’ check `now - last_message_start_ >= requestTimeout`.
- Otherwise (still reading headers) â†’ check
  `now - last_message_start_ >= headersTimeout`.
- Stop iterating as soon as one parser is *not* expired (the set
  ordering guarantees nothing later is older).

`headers_completed_` is set inside the `on_headers_complete` llhttp
callback.

### Periodic check on the JS side

`libjs-node/_http_server.js` already contains:

```js
this[kConnectionsCheckingInterval] =
  setInterval(checkConnections.bind(this), this.connectionsCheckingInterval).unref();

function checkConnections() {
  if (this.headersTimeout === 0 && this.requestTimeout === 0) return;
  const expired = this[kConnections].expired(this.headersTimeout, this.requestTimeout);
  for (const parser of expired) parser.onTimeout();
}
```

Today `expired()` always returns `[]`, so the interval is harmless but
also useless. After this plan it becomes load-bearing.

## Design

### Required additions to `Parser`

```cpp
class Parser {
  // ...
  ConnectionsList *connectionsList_ = nullptr;
  uint64_t         lastMessageStart_ = 0;
  bool             headersCompleted_ = false;   // already exists
};
```

`headersCompleted_` already exists in our code; we just need to make sure
it is consulted from `Expired()` (defined on `ConnectionsList`, but it
needs to peek into `Parser::headersCompleted_` and
`Parser::lastMessageStart_` â€” declare `ConnectionsList` a friend of
`Parser`, or add public accessors).

### Required additions to `ConnectionsList`

```cpp
class ConnectionsList {
  // Replace std::vector<napi_ref> parsers_ with two ordered sets keyed
  // on (lastMessageStart_, parser*). napi_ref is per-parser; the
  // ordered key is the Parser* (with its current timestamp pulled from
  // the field at insertion time). To keep the set valid we Pop before
  // mutating the timestamp and Push after.

  struct ParserComparator {
    bool operator()(const Parser *a, const Parser *b) const {
      if (a->lastMessageStart_ != b->lastMessageStart_)
        return a->lastMessageStart_ < b->lastMessageStart_;
      return a < b;
    }
  };

  std::set<Parser*, ParserComparator> all_;
  std::set<Parser*, ParserComparator> active_;
  // Per-parser napi_ref so JS's `connections[i].socket.destroy()` keeps
  // working. Map from Parser* so we can free the ref on Pop.
  std::unordered_map<Parser*, napi_ref> refs_;
};
```

JS-callable shape stays the same (`all`, `idle`, `active`, `expired`),
all returning arrays of parser objects.

### Required additions to lifecycle

- `ParserInitialize` (already partly done in the minimal fix):
  - Stash list pointer.
  - Set `lastMessageStart_ = uv_hrtime()`.
  - `list->Push(parser); list->PushActive(parser);`
- `Parser::sOnMessageBegin` trampoline:
  - Existing logic stays.
  - **Before** any field reset:
    `if (connectionsList_) { Pop; PopActive; }`
  - Update `lastMessageStart_ = uv_hrtime()`.
  - **After** reset:
    `if (connectionsList_) { Push; PushActive; }`
- `Parser::sOnMessageComplete` trampoline:
  - **Before** anything that touches `lastMessageStart_`:
    `if (connectionsList_) { Pop; PopActive; }`
  - `lastMessageStart_ = 0;`
  - `if (connectionsList_) { Push; }` // not active anymore
- `ParserRemove` (already partly done):
  - `if (connectionsList_) { Pop; PopActive; connectionsList_ = nullptr; }`
- `Parser::~Parser` / `pointerCb`:
  - Defensive: if `connectionsList_` is still set (Remove was not
    called), `Pop`+`PopActive` so we don't leave a dangling pointer in
    the set. Mirror Node's invariant.

### Required additions to `Expired`

```cpp
napi_value expired(napi_env env, double headersTimeout, double requestTimeout) {
  uint64_t now = uv_hrtime();
  uint64_t headersTimeoutNs = headersTimeout * 1e6;  // ms -> ns
  uint64_t requestTimeoutNs = requestTimeout * 1e6;

  std::vector<Parser*> expired;
  for (Parser *p : all_) {
    uint64_t threshold = p->headersCompleted_ ? requestTimeoutNs
                                              : headersTimeoutNs;
    if (threshold == 0) continue;  // disabled
    if (now - p->lastMessageStart_ < threshold) break;  // ordered set
    expired.push_back(p);
  }
  // Build JS array of parser objects from refs_.
  // ...
}
```

### Constants

The JS side passes timeouts in **milliseconds** (per Node's API).
`uv_hrtime()` returns nanoseconds. Convert at the boundary, not in the
middle of the comparison.

## Implementation steps (each independently testable)

### Step 1 â€” Make `Push`/`Pop` work

Replace `std::vector<napi_ref> parsers_` with the dual-set + map model.
Keep the existing `push`/`remove` JS-callable surface; rewrite to keep
the sets consistent.

**Test**: existing `test-http.js` test 12 (the one we just added)
keeps passing.

### Step 2 â€” Track the active subset

Add `pushActive`/`popActive`. Wire them at `Initialize`,
`on_message_begin`, `on_message_complete`, `Remove`. Update `idle()` to
return `all_ \ active_` and `active()` to return `active_`.

**Test**: new `test-http-close-idle.js`. Spin up a server with two
keep-alive clients. Have one mid-request (server is reading body
slowly), the other idle between requests. Call
`server.closeIdleConnections()`. Verify the idle connection's socket
emits `'close'` and the active one keeps its socket open (FIN not
received). Then complete the active request, repeat
`closeIdleConnections()`, and watch *that* socket close.

### Step 3 â€” Track `lastMessageStart_`

Set it at Initialize, reset at on_message_begin, clear at
on_message_complete. Always Pop the parser out of both sets *before*
mutating, Push after. Add an invariant assert (in `HERMES_SLOW_DEBUG`
builds) that the set's ordering is preserved across each lifecycle
transition.

**Test**: not directly observable, covered by Step 4.

### Step 4 â€” Implement `expired()`

Per spec above.

**Test**: new `test-http-headers-timeout.js`. Server with
`headersTimeout = 100ms`. Open a raw TCP socket, send a partial request
("GET / HTTP/1.1\r\nHost:") and stop. Wait 200ms. Verify the server
emits `'requestTimeout'` (or whichever event Node fires) and destroys
the socket. Mirror with `requestTimeout` and a partial body.

### Step 5 â€” Documentation + cleanup

- Replace the "simplified" disclaimer in the binding header with a
  description of the actual semantics.
- Add a paragraph to `doc/DESIGN.md` (or wherever HTTP infra is
  described) explaining the active/idle invariant so future contributors
  don't reintroduce the gap.

## Tests

- `test-http-close-idle.js` (new) â€” Step 2 verification. Two parallel

- `test-http-headers-timeout.js` (new) â€” Step 4 verification. Raw

## Process change to prevent regressions

Going forward, when porting a binding from Node:

1. Enumerate every JS-side method on the binding's primary export (read
   the JS file that consumes it). Don't trust internal-comment
   annotations like "TODO" or "simplified" to be revisited.
2. For each method, write at least one lit test that exercises a
   non-trivial code path on it â€” even when our own use of the binding
   doesn't go through that method.
3. Whenever a binding's C++ field is `// unused for now`, file a
   follow-up; agent-driven ports drop these on the floor by default.

The current incident took multiple debugging sessions and a deep dive
through the JS stream-destroy / socket-close chain to localize, all
ultimately for a single `args[4] = connectionsList (optional, unused
for now)` line in a 1300-line commit. The remediation cost is
disproportionate to the cost of writing a one-method coverage test up
front.
